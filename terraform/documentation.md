Think of these five files as having distinct roles in a single system. They do not work independently — each one depends on the others. Here is the mental model before the detail:

**`variables.tf`** — declares what inputs the system needs, like a form with blank fields
**`terraform.tfvars.example`** — fills in those fields with actual values for your environment
**`providers.tf`** — tells Terraform which cloud platform to talk to and how to authenticate
**`main.tf`** — the orchestrator — calls each module with the values it needs
**`outputs.tf`** — prints the results after everything is built, so you can see what was created

---

**`variables.tf` — the declaration of inputs**

This file does not set any values. It only declares that certain inputs exist and what type they must be. Think of it as defining the contract: "this project requires a `project_id`, a `region`, an `environment`" and so on.

Some variables have no default, which means they are mandatory — Terraform will refuse to run without them. `project_id`, `region`, `environment`, `project_name`, and `organization_prefix` all fall into this category. Others like `labels`, `force_destroy_buckets`, and `notification_email` have defaults, so they are optional.

---

**`terraform.tfvars.example` — the values**

This is where the declared variables get their actual values. The `.example` suffix means this is a template — you copy it, rename it to `terraform.tfvars`, and fill in your real values. The reason it is kept as an example in Git rather than the real file is security: your real `terraform.tfvars` contains project-specific values and should never be committed to a shared repository.

For your project, you would set:

```
project_id           = "manish_sandpit"
region               = "australia-southeast1"
environment          = "dev"
project_name         = "hackathon"
organization_prefix  = "intelia"
notification_email   = "your-email@example.com"
```

That `name_prefix` you see in `main.tf` — `"${var.organization_prefix}-${var.project_name}-${var.environment}"` — would resolve to `intelia-hackathon-dev`. This prefix is stamped on every resource name so you can identify at a glance that a bucket or dataset belongs to this project.

---

**`providers.tf` — the connection to GCP**

Terraform is cloud-agnostic by design — the same tool can provision AWS, Azure, or GCP. The provider file is what tells it which cloud to use and where to find the plugin that knows how to talk to that cloud. The `google` provider here is essentially a large library of GCP-specific resource types. The `~> 5.0` version constraint means "any version in the 5.x family" — it will accept 5.1 or 5.9 but not 6.0, which protects you from breaking changes in a major version upgrade.

The two lines at the bottom — `project = var.project_id` and `region = var.region` — are what connect every resource Terraform creates to your specific GCP project. Without these, Terraform would not know where to deploy anything.

---

**`main.tf` — the orchestrator**

This is the most important file. It calls seven modules — `project`, `storage`, `bigquery`, `iam`, `networking`, `dataplex`, `monitoring` — and passes each one the values it needs to do its work. Each module is a self-contained folder of Terraform code that knows how to create a specific set of resources. `main.tf` does not know the internal details of each module — it just calls them and wires their inputs together.

Two things worth noting:

The `locals` block at the top computes derived values that are used repeatedly — the `name_prefix` and the `common_labels` map. Rather than repeating the same string expression in seven places, it is computed once and referenced everywhere. If you need to change the naming convention, you change it in one place.

The `depends_on` lines are dependency declarations. `module.storage` has `depends_on = [module.project]` because the storage bucket cannot be created until the Cloud Storage API is enabled — which is what `module.project` does. Terraform uses these to determine the order in which modules are executed. Without them, Terraform might try to create a bucket before the API is enabled and fail.

---

**`outputs.tf` — the results**

After `terraform apply` completes, Terraform prints everything declared in this file to the terminal. These are not just cosmetic — they are the actual names and IDs of the resources that were just created, which other tools need to reference. For example, the Cloud Run Validator needs to know the exact name of the Pub/Sub topic to publish to — that name is generated by Terraform during apply and surfaced here as `pubsub_topics.pipeline_coordinator`. The ADK agents, Cloud Build pipeline, and Dataform configuration all consume values that originate from these outputs.

---

**How they work together — the sequence**

```
terraform.tfvars        ← you fill in your values here
      ↓
variables.tf            ← Terraform reads the declared types and validates your values
      ↓
providers.tf            ← Terraform loads the GCP plugin and connects to manish_sandpit
      ↓
main.tf                 ← Terraform calls each module in dependency order,
                           passing the resolved variable values
      ↓
outputs.tf              ← Terraform prints the names and IDs of everything created
```

Running `terraform plan` runs this entire sequence but stops before making any changes — it shows you exactly what would be created, changed, or destroyed. Running `terraform apply` does the same and then executes the changes. That is why the implementation plan calls for reviewing the plan output before applying — it is your last check before real resources are created.

var. is Terraform's syntax for referencing a value that was declared in variables.tf and supplied via terraform.tfvars.
The three pieces work together like this:
variables.tf declares that a variable called project_id exists:
hclvariable "project_id" {
  type        = string
  description = "The GCP Project ID where resources will be deployed"
}
terraform.tfvars supplies the actual value:
hclproject_id = "manish_sandpit"
Anywhere else in your Terraform code — providers.tf, main.tf, outputs.tf — var.project_id is how you read that value back:
hclproject = var.project_id   # resolves to "manish_sandpit"
